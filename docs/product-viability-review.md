# MiaoTarot 产品可玩性、复玩与付费评审

更新时间：2026-07-18

本报告综合了代码与 Git 历史、桌面/手机竖屏/低高度横屏实际走查、当前埋点与部署状态，以及[商业案例研究](commercial-reference-research.md)。它是证据审计和产品判断，不是用户研究结果。

## 最终判断

| 问题 | 当前判断 | 置信度 | 原因 |
| --- | --- | --- | --- |
| 有人会点进来吗 | 会，猫 + 塔罗 + 高完成度视觉有明显吸引力 | 中 | 本地启发式评测，没有线上入口数据 |
| 有人会完成一次吗 | 会，但默认三牌路径对轻用户偏重 | 中 | 交互闭环清楚；动作多、嵌套滚动和长页面增加流失 |
| 有人会再玩吗 | 少部分会，整体动机尚弱 | 中 | 有今日牌和本地历史，但没有连续性、提醒、比较或回访测量 |
| 有人会分享吗 | 工程能力强，真实触发大概率偏低 | 中高 | 分享入口离最后翻牌很远，且默认暴露用户问题 |
| 有人现在会付费吗 | 很少，且没有任何证据 | 高 | 没有价格、商品、支付、付费访谈或长期价值 |
| 值得继续吗 | 值得，但下一阶段应验证分发与复访，而不是继续堆功能 | 中高 | 已有可玩的完整闭环，最大风险已经从“能否做出”变成“是否有人要” |

如果一定要给启发式分数：首次开始约 **6.5/10**，完成一次约 **5.5/10**，复玩约 **3.5/10**，分享动机约 **3/10**，当前付费意愿约 **1.5/10**。这些不是转化率预测，只用于表达相对强弱。

## 证据边界

仓库能证明产品已经可玩，不能证明真实用户愿意玩。当前没有可识别的 canonical 生产网址；`https://tarot.pages.dev` 实测仍是旧站，README 仍以 `https://your-domain.example` 作为生产示例。Wrangler 远端数据查询又因缺少 `CLOUDFLARE_API_TOKEN` 无法进行。

现有事件只有：

- `reading_started`
- `reading_completed`
- `daily_reading`
- `share_copied`、`share_image`、`share_result`
- `llm_requested`、`llm_succeeded`、`llm_failed`

它们在 [`productAnalytics.ts`](../site/src/domain/productAnalytics.ts) 中定义，后端只按“日期 × 事件 × variant”聚合到 [`0002_product_events.sql`](../migrations/0002_product_events.sql)。没有匿名 session、来源、share token 或 cohort，所以无法计算真实漏斗、D1/D7、分享回流和用户级复玩。

因此报告中的“会玩/会付费”是产品推断；只有上线后的行为和真实支付能验证。

## 当前产品究竟是什么

仓库最初的策略很清楚：不做通用 AI 塔罗，而做一个以猫咪状态为情绪接口的 meme-native 自我观察玩具，路径是“分享玩具 → 每日仪式 → 社交模式 → `xxxTarot` 主题平台”，见 [`product-strategy.md`](product-strategy.md)。

最近的实现则逐渐转向“认真塔罗 + 猫画风”：标准牌名、传统正逆位和长牌义被放到前台，猫 archetype 和 meme 语言变弱。这个方向更可信，却也更容易落入已有塔罗应用的同质化竞争。

真正需要验证的不是哪种内部观点更正确，而是两个明确命题：

- **A：猫状态翻译器。** 用户先获得“这只猫就是我”的身份共鸣，塔罗依据在详情层；
- **B：可信的猫咪塔罗。** 用户先获得标准牌义和牌阵解释，猫只负责降低门槛。

当前代码和文档在 A/B 之间摇摆。下一步应让两套首页和分享卡面对真实流量，而不是继续凭偏好来回改文案。

## 首次用户路径评审

### 首屏：好看、克制，但仍像一个开发项目

强项：

- “抽一张猫咪塔罗，换个角度看清问题”直接、低风险，没有恐吓和宿命化；
- 主视觉完成度高，猫与三张牌一眼可懂；
- `开始抽牌` 和 `今日一牌` 都无需登录；
- 有默认问题，不填表也能开始。

问题：

- 首屏长段落先讲 15 世纪纸牌史，情绪钩子被知识解释稀释；
- 审计基线中的 `GitHub`、`Research`、页脚技术栈和 `Implementation plan` 让用户把它识别成 demo；本次已做文字降噪，但公开链接仍然指向开发文档；
- `复制一句介绍` 和两个核心体验入口同级，抢占第一次决策；
- 自然滚动顺序是 hero → 大段塔罗科普 → 牌桌。手机上牌桌位于约 2.35 屏之后，虽然 CTA 会锚定跳过，但页面默认叙事仍然是“先上课”。

### 设置：默认值有帮助，选项仍过多

第一次就出现牌数、内容包、主题、逆位、问题、快捷问题和音效。默认值可以直接开始，但用户仍要理解“内容包”“逆位”和为何需要从 1–5 中选择。

更合理的轻重分层是：

- 默认只显示“问题 + 开始洗牌”；
- 第一次默认单张；
- 牌数、牌组、逆位放入“更多设置”；
- 用户完成一次后再推荐三牌或关系牌阵。

这与 Zappallas 的“轻量内容 / 粉丝深度内容”分层一致，也符合 [Fogg 行为模型](https://www.behaviormodel.org/home)中同时降低能力门槛、保留明确 prompt 的思路。

### 洗牌和选牌：仪式感是核心资产

洗牌、声音、完整牌堆、亲手选牌、放上桌和逐张翻牌构成了比“点击随机结果”更有参与感的闭环。身份和正逆位在洗牌时固定，也减少“结果是最后一刻改出来”的不信任。

但默认三牌需要约 8 个显式动作。78 张背面完全相同，用户并没有信息去做“更好选择”；在手机上又形成页面滚动 + 牌堆内部滚动。仪式感和选择疲劳之间需要更精确的平衡。

建议保留完整牌堆作为高级仪式；默认单张可用扇形、滑动停牌或有限可视窗口，让用户完成“我亲手选了”即可，不要求浏览全部 78 张。

## 横屏与竖屏

### 手机竖屏

优点：设置区单列清楚，无页面横向溢出；牌堆为 5–6 列，卡面可点。

问题：默认三牌完成后的交互牌桌约 2,208px，紧接着重复的 `ReadingResult` 约 3,213px；整个页面实测约 8,572px。第三张翻开时，分享区域仍在约 4,400px 之后，相当于还要跨约 5.7 个 767px 手机屏。

### 低高度横屏

在 nominal 844×390、实际 CSS viewport 约 767×354 的环境下，页面刚好越过 760px 手机断点，使用桌面 hero。长描述与右侧图片叠在一起，主 CTA 位于首屏下方；设置区又因为 Mantine `md` 未触发而保持单列，开始按钮要滚近两屏才出现。

只有牌堆 viewport 针对低高度横屏做了限制，hero、设置和结果都没有。横屏不能只按宽度判断，应同时对 `orientation: landscape` 与 `max-height` 做紧凑布局。

## 结果、分享与复玩

### 结果重复消耗了情绪高潮

交互牌桌翻牌时已经显示图片、标题、释义和详情；随后 `ReadingResult` 又展示主牌摘要和每张牌完整卡片。相同信息连续出现两次，使移动端极长，并把真正的行动入口推到页面底部。

完成最后一张牌后，理想的同屏内容应只有：

1. 一张核心猫牌；
2. 一句“这就是我”的 caption；
3. 一个今天能做的小动作；
4. `分享`、`再来一次`、`继续深挖` 三个按钮。

传统牌义、正逆位和逐牌详情可以折叠，不应该丢失；但它们不需要重复成第二套完整结果。

### 分享基础设施强，内容动机和隐私弱

已经有系统分享、复制文案、PNG、QR 和可还原结果的 URL，这是值得保留的工程资产。

但分享文案和 URL 默认带上用户原问题。关系、工作和心理状态都可能敏感，本次已补发布前提示，但仍没有“隐藏我的问题”开关。分享卡又偏标准塔罗摘要，缺少“这只猫太像我”的身份表达。

真正可传播的结果应该：

- 默认隐藏问题，用户显式同意才包含；
- 不剧透好友的体验，但能表达自己的状态；
- 让接收者落地后直接看到分享结果和“一起抽”，而不是从 hero 和科普重新开始；
- 用 share token 归因“分享访问 → 完成 → 再分享”的增长循环。

### 复玩有原语，没有闭环

当前有 `今日一牌`、`同玩法再洗`、`换玩法` 和本地最近 8 次历史。但这些能力没有形成连续价值：

- 完成后的“同玩法再洗”在牌桌顶部，手机用户要向上找；
- 历史只可回看，没有对比、笔记、变化或周报；
- 今日牌按日期和内容包确定，同一天所有用户会得到同一张。它适合叫“共同今日主题”，不适合暗示个人随机抽取；
- 无账号、跨设备、提醒或回访 cohort。

不要先用 streak 强迫每日使用。先验证用户的自然频率究竟是每日、每周还是在具体事件发生时；Reforge 的[低频产品 ICED 框架](https://www.reforge.com/blog/iced-theory-growing-infrequent-products)强调产品节奏必须匹配自然需求。

## 信任审计

当前建立信任的部分：

- 明确“不预测命运”，以自我观察定位；
- 科普历史并链接博物馆来源；
- 抽牌核心不依赖 AI，AI 未开放也不影响基础结果；
- 服务端 AI prompt 已禁止宿命化，并对高风险问题设置专业支持边界；
- 本地历史和聚合事件的数据收集相对克制；
- 洗牌时确定牌与正逆位，逻辑可解释。

当前破坏信任的部分：

- 简体界面混入繁体标准牌名和长段牌义，部分 advice 像未经编辑的数据 dump；
- “猫语”标签下经常只是传统牌义，品牌承诺和结果人格不一致；
- 本次已补用户可见安全边界，但仍缺少隐私、条款、联系方式和运营主体；
- 分享默认携带敏感问题；
- 公开的开源与产品说明仍指向开发文档，商业产品信任信息不足；
- 若未来使用知名猫 meme 商业化，多数素材的商业改编许可仍未解决，见 [`famous-cat-meme-source-register.md`](famous-cat-meme-source-register.md)。

## 用户是否愿意付费

### 当前答案：大概率不愿意

代码中没有商品、价格、支付、订阅、订单或 pay-intent 事件。用户已经免费获得抽牌、基础结果、历史、分享和 AI 入口，也没有账户、长期模式、个人资产、真人关系或强创作者内容可收费。

更重要的是，Labyrinthos 已经证明“完整、好用、用户多”仍可能只有不到 1% 付费。MiaoTarot 现在不能用完成度推导支付意愿。

### 不要按“塔罗用户”笼统分群，要按任务分群

| 用户此刻要完成的任务 | 自然频率 | 免费入口 | 合适收入 | 不合适收入 |
| --- | --- | --- | --- | --- |
| 找一个好玩、像自己的结果 | 偶发/社交触发 | 单张猫状态卡 | 限定皮肤、品牌主题、收藏资产 | 月订阅 |
| 整理一个具体问题 | 事件触发 | 基础牌义和一个行动提示 | 一次深读、credits、有限追问 | 在抽牌前硬墙 |
| 建立反思习惯 | 每周或每日 | 今日主题、简单历史 | 日记、趋势、人工内容订阅 | 只卖无限 AI 字数 |
| 学会读牌 | 长周期 | 免费牌义、练习 | 课程、练习社群、工具 | 按次预测 |
| 找可信的人帮忙 | 高意图、低频 | 顾问发现与首次匹配 | 按时间/场次咨询 | 匿名 AI 冒充真人 |

这五种任务需要不同产品和商业模型。当前 MiaoTarot 最适合前两种，也具备向第三种发展的原语；不应因为 AstroTalk 盈利，就把第五种重市场模型当成当前答案。

### 用户可能为什么付费

按与当前产品的匹配度排序：

1. **一次性深度解读或 credits。** 免费基础结果之后，针对当前具体问题提供结构清楚的一次性深读、同一组牌的有限追问；
2. **猫牌日记和周/月回看。** 跨设备历史、自己的解释、行动结果、重复牌与主题趋势；只有真实复访后才做订阅；
3. **限定数字/实体牌和个人猫资产。** 视觉差异比普通 AI 文本更强，也更接近已有投入；
4. **主题报告。** 关系、工作、创作者或项目年度报告，适合一次性购买；
5. **课程、练习和社区。** 只有当品牌建立内容权威后，才参考 Biddy Tarot 的学习阶梯；
6. **真人顾问。** 已有市场证明客单高，但运营最重，不应成为当前路线。

### 最可能的早期收入不是消费者订阅

仓库已经有主题化 tarot 基础和 ShipTarot。没有海量消费者时，更现实的早期收入测试是：

- 给品牌、创作者或社群制作可分享的主题抽卡页；
- 预售限定数字/实体猫牌；
- 在免费结果后测试一次性深读，而不是直接做月费。

这三种路径分别利用现有主题引擎、78 张视觉资产和结果上下文，且不要求先证明每日留存。

## 建议的增长与收入系统

不要把所有目标混成一个漏斗。MiaoTarot 需要三台不同的引擎：

1. **获得引擎：** 免费、快速、可分享的身份结果；
2. **返回引擎：** 今日/每周有限仪式 + 历史、笔记和跨时间洞察；
3. **收入引擎：** 具体问题深度、收藏资产、持续内容或企业定制。

当前获得引擎的底层能力已经具备，但结果距离分享太远；返回引擎只有今日牌和 8 条本地历史；收入引擎尚未存在。

增长应设计成[可重复的 loop](https://www.reforge.com/blog/growth-loops)：

`完成阅读 → 生成隐私安全的猫状态卡 → 好友打开共同结果 → 好友完成自己的阅读 → 再产生分享`

North Star 不应是访问量或抽牌总数，而是“在自然周期内完成有价值阅读并再次回来的人”。在确认自然频率前，建议先用 **7 天内完成第二次阅读的匿名用户数** 作为候选核心指标。

## 90 天验证计划

### 阶段 0：先建立可测的生产事实

- 发布当前 `main` 到唯一 canonical URL；
- 验证 D1 binding、counter、product-event 和生产 smoke；
- 加匿名 `session_id`、首访 `source`、`share_token`、页面/按钮曝光；
- 不存用户问题全文；明确 retention 和删除策略；
- 建立 `landing → hero CTA → started → selected → placed → final flip → result CTA` 漏斗。

没有这一步，所有“用户愿意”都只是内部感觉。

### 阶段 1：30–50 人任务测试

分三类各招至少 10 人：猫/梗用户、塔罗用户、轻度怀疑者。让用户独立完成：

1. 第一次抽牌；
2. 找到最有用的一句话；
3. 决定是否愿意分享；
4. 第二天或一周内是否愿意再来。

记录开始/完成/分享和关键卡点，并只问三个开放问题：你以为它会给你什么、哪一刻最有感觉、为什么不想分享或回来。

当至少约 40 名用户在两周内使用过核心体验两次后，再使用 [Superhuman 的 PMF 调查方法](https://review.firstround.com/how-superhuman-built-an-engine-to-find-product-market-fit/)：问“如果不能再使用，你会怎样”，并分析“非常失望”人群真正看重的价值。40% 只能作为方向性信号，不是发布前必须追逐的通用行业标准；没有重复使用的人不应被混进样本。

### 阶段 2：两个定位实验

- A 版：`把你现在的状态，翻译成一只猫`，分享卡以猫 archetype 为主；
- B 版：`可信的猫咪塔罗`，分享卡以标准牌义为主。

同时比较 hero CTA、完成、最后翻牌到分享、分享回流和 7 日二次阅读。不要只看点击率。

### 阶段 3：付费意图而非问卷

完成基础结果后，随机展示三个透明的真实 offer：

- `¥6.9–12.9 一次深读`：说明字数、内容、是否 AI；
- `¥18–38 主题报告/数字牌组`：一次购买；
- `本周猫牌总结`：先加入候补，只有复访成立后再测试订阅。

点击后明确告诉用户“功能尚在测试，可加入候补”，不收卡、不伪造结账。记录 offer impression、详情点击和留资；真正判断支付意愿时应尽快升级为小额真实付款或可退款预售。

### 阶段 4：手工 concierge 验证价值

不要先开发完整 AI 对话和周报。挑 10–20 个愿意付费或高意向用户，手工交付：

- 一次结构化深读；
- 一周后的回看与模式总结；
- 一张个人猫主题结果卡。

比较哪一种被使用、被保存、被转述和再次购买，再决定自动化方向。

## 决策门槛

以下是团队决策阈值，不是行业基准：

- 开始后完成率若仍低于 60%，先修路径，不做付费；
- 最后翻牌后能看到结果 CTA 的用户中，分享意图低于 8%，先修结果内容和隐私；
- 7 天二次阅读低于 10%，不做订阅；
- 付费详情点击低于 5%，先换价值主张，不接支付；
- 有至少 10 个真实小额购买或可退款预售，并出现重复使用，再讨论订阅；
- 若 B2B 定制在 20 个精准触达中拿不到 3 次有效沟通，也不要把“主题平台”当近期收入假设。

阈值必须在实验前写下，防止看到任何正向数字后临时解释成功。
