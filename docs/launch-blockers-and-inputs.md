# 首次上线阻塞与输入清单

更新时间：2026-07-24

这份文档回答三个问题：

1. MiaoTarot 当前究竟被什么阻塞；
2. 哪些问题可以由仓库内的实现和验证解决，哪些必须获得外部输入；
3. 外部输入应该去哪里找、以什么形式提供，以及什么才算完成。

它是上线判断的入口清单，不重复 [产品与验证方向](product.md)、[LLM 交互与上线架构](llm-interaction.md) 和[图片生成与资产契约](image-generation-contract.md)里的详细规范。

## 先说结论

MiaoTarot **已经具备公开测试所需的技术链路**，不是被 Qwen、流式、多轮、D1 或 Cloudflare 部署阻塞。2026-07-24 对 `https://tarot-31o.pages.dev` 的强制生产 smoke 已确认当前构建、Pages Functions、Analytics Engine、公开计数、可选 D1 会话备份和 `qwen3.7-plus` 均可用；内容校验也通过。

当前差距分为三层：

| 目标 | 当前状态 | 真正阻塞 |
| --- | --- | --- |
| 继续发布公开测试版 | 技术链路已具备 | 只需完成当前移动端单牌可读性修复、端到端验证并按发布流程重新部署；不需要新的产品输入 |
| 把它称为“完成验证的正式首版” | 尚不具备 | 缺少 5–8 位非团队真实用户的形成性测试，以及一份明确的上线主张 |
| 扩大获客、强调“真实热梗牌组”或开始收费 | 尚不具备 | 缺少可归因流量、逐卡内容审校、可商用 meme 来源和真实支付证据 |

因此不能把所有缺口都塞进“上线阻塞”。公开测试可以继续；真实用户价值、meme-native 品牌和收费准备度仍待验证，宣传时不能把它们写成已经成立的事实。

## 阻塞清单

### A. 当前发布必须完成

| 项目 | 当前证据 | 需要的输入 | 输入去哪里找 | 完成标准 |
| --- | --- | --- | --- | --- |
| 手机端单牌阅读可读性 | 旧布局把完整牌面和说明左右挤在窄屏中，牌面过小且文字有越框风险，sticky 牌条还会遮挡阅读 | 不需要用户补内容；这是仓库内的交互与 CSS 问题 | 以 `320px` 和 `390×844` 真实路径截图、DOM 尺寸和触摸操作为准 | 每张正在解释的牌独占全宽视觉行；牌面可点击翻到牌义；详情可展开；文字不越框；无 sticky 遮挡；两种手机视口 E2E 通过 |
| 发布产物一致性 | 线上技术链路当前可用，但每个新 commit 都必须重新证明构建与部署一致 | 不需要新的产品内容；部署者需要已有 Cloudflare 权限 | `AGENTS.md`、[工程手册](engineering.md#cloudflare-pages-发布) 和发布脚本 | `npm run verify:launch` 通过；从最终 commit 干净构建；发布后生产 HTML 的资源文件名与本地 `v1/index.html` 一致；生产 smoke 与 E2E 通过 |

### B. 正式首版的产品判断必须补齐

| 项目 | 为什么受阻 | 需要提供什么 | 怎么获得 | 完成标准 |
| --- | --- | --- | --- | --- |
| 上线主张与边界 | 仓库可以证明功能可用，不能证明目标人群、留存或“AI 更懂用户” | 一张产品负责人确认的上线卡：上线类型、核心人群、第一价值、允许公开声称什么、明确不声称什么 | 这是产品决策，不能靠搜索或模型代填。使用下方模板由负责人确认 | 官网、商店文案、发布帖与产品实际能力一致；不把小样本、测试事件或推断写成用户事实 |
| 真实用户形成性测试 | 自动化只能证明链路工作，不能证明普通用户看得懂重点纠正、单牌解读和后续追问 | 5–8 份经过同意、去标识化的非团队记录；不需要姓名、问题原文或聊天全文 | 从猫咪兴趣用户、塔罗好奇者、轻度怀疑者和有塔罗经验者中招募；按 [5–8 人 pilot 执行卡](llm-interaction.md#58-人真实用户-pilot-执行卡)主持 | 至少 5 人 `non_team_confirmed=yes` 且完成阅读；记录主要困惑、是否理解可纠正、解释是否有帮助；根据重复问题修正后再判断首版 |
| 内容审校负责人 | 大阿卡纳和完整 RWS 对照已形成基础，但 56 张小阿卡纳仍大量复用牌阶模板，不能支持“逐卡深度专业内容”或付费深读主张 | 对每张卡的正/逆位边界、猫语、现实例子、高风险表达与牌位适配做编辑签核 | 标准骨架先读仓库内 `@cometpisces/tarot-kit` 映射、`site/src/domain/miaoTarot.ts`、`site/src/domain/miaoMinorArcana.ts`；再由熟悉塔罗且能做中文编辑的人复核，不从营销号拼接牌义 | 每张目标卡有审校记录、修改原因和签核人；关系、工作、选择三个高频主题先覆盖；未完成前只称“基础学习图鉴”，不称“专业逐卡深读” |

### C. 扩大传播或收费前必须补齐

| 项目 | 为什么受阻 | 需要提供什么 | 怎么获得 | 完成标准 |
| --- | --- | --- | --- | --- |
| 可归因的目标用户流量 | 当前明确 external 流量很少且全部为 direct，不能判断谁在使用、为何分享或为何回来 | 2–3 个可追踪内容入口、对应受众假设和至少一个自然回访周期 | 使用带来源参数的发布链接或 share token；内容入口可来自创作者账号、猫咪社群或塔罗社群，但不得购买虚假流量 | internal smoke 被排除；能分别看到入口 → 完成阅读 → 分享 → 7 日二次阅读，且样本量足以描述而不是猜测 |
| “真实热梗牌组”来源授权 | 当前内容校验为 `0 verified meme sources`、`7 legal fallbacks`、`3 research-only`；合法 fallback 可以上线，但不能冒充可识别热梗来源 | 原始发布页、作者、许可页或书面商用改编授权、下载地址、检索日期、尺寸、SHA-256、署名与衍生要求 | 优先找创作者原帖和明确标注 `CC BY` / `CC BY-SA` 的原始资源；没有公开许可就联系作者取得书面许可。聚合站、水印图、二次搬运和“网上都在用”都不构成授权 | manifest 状态为 `verified-meme-source`，来源和授权可复核，侧对侧评审通过；否则继续使用原创/合法 fallback，并删除“真实热梗牌组”主张 |
| 付费与收入证据 | 当前只有创作者支持入口，没有订单、金额、支付回调、交付或退款状态 | 明确商品、价格、交付范围、退款规则，以及支付服务商的订单/到账证据 | 先做不扣款的真实商品详情测试；达到团队预设意向线后，再从支付服务商后台或回调记录订单，不从评分数、点击或赞赏码曝光反推收入 | 至少出现 10 个真实小额购买或可退款预售，并完成交付与退款记录；在此之前不做订阅收入结论 |

## 需要负责人提供的最小输入包

当前 UI 修复、测试、commit 和 push 不需要等待这些输入。若下一步要宣布“正式首版”或开始对外推广，负责人只需先补下面三项：

### 1. 上线卡

```yaml
launch_type: public_beta | official_free | paid_pilot
primary_audience: 一句话描述谁会在什么时刻打开它
first_value: 用户完成第一张牌后应得到什么
allowed_claims:
  - 可以公开声称的事实
forbidden_claims:
  - 还没有证据、不能公开声称的结论
canonical_domain: 唯一生产网址
product_owner: 负责最终取舍的人
```

如果只继续公开测试，建议默认使用：

```yaml
launch_type: public_beta
primary_audience: 喜欢猫咪或轻网络文化、对塔罗好奇、正卡在工作关系或选择问题里的手机用户
first_value: 一分钟内从一张亲手翻开的牌得到一个可理解、可追问的新视角
allowed_claims:
  - 支持 78 张牌、正逆位、基础学习图鉴、流式 AI 解读和浏览器内会话恢复
forbidden_claims:
  - AI 更懂用户
  - 已验证能改善决策或心理状态
  - 已拥有大量用户、留存或付费订阅
  - 全套牌面均来自真实热梗
canonical_domain: https://tarot-31o.pages.dev
```

### 2. Pilot 招募输入

提供 5–8 个愿意参加 10–15 分钟手机测试的非团队联系人，或由负责人指定一个可以公开招募的受众渠道。仓库只保存 P01–P08 这类编号和去情境化观察，不保存：

- 真实姓名与联系方式；
- 用户问题原文；
- 完整聊天；
- 医疗、法律、投资、危机或第三方隐私内容。

如果找不到完整 8 人，先按四种视角各找至少一人：猫咪兴趣、塔罗好奇、轻度怀疑、有塔罗经验；其余名额补当前最核心人群。朋友可以参加，但项目成员和已经熟悉实现细节的测试者不算非团队样本。

### 3. 内容与图片输入

内容审校需要的是“可核对的编辑意见”，不是再生成一批长文。每张牌使用统一格式：

```yaml
card_id:
orientation: upright | reversed
traditional_anchor:
miao_copy:
real_world_example:
high_risk_boundary:
review_status: needs_change | approved
reviewer:
```

图片来源需要的是授权证据，不是只有图片文件。使用[图片生成与资产契约](image-generation-contract.md#来源与授权)中的 manifest 字段；如果没有授权，最安全的输入就是具体猫咪行为描述和塔罗锚点，由 MiaoTarot 生成原创画面，不模仿或复刻知名 meme 原图。

## 输入查找顺序

不要先上网搜一堆零散素材。按下面顺序找，能显著减少重复劳动和来源歧义：

1. **先查仓库事实。** 产品结论读 `docs/product.md`；AI、多轮、存储和生产门读 `docs/llm-interaction.md`；图片来源读两个 manifest；运行 `npm run verify:content` 和 `npm run verify:launch`。
2. **再查生产事实。** 使用强制参数运行 `smoke:production`、真实 provider 的 `smoke:llm` 和生产手机 E2E。Secret 只写入 Cloudflare，不粘贴到 issue、聊天或文档。
3. **再找只能由人产生的输入。** 上线主张由产品负责人签；内容由编辑/塔罗审校人签；可用性由非团队参与者产生；支付结果以 provider 账单和回调为准。
4. **最后才做外部素材研究。** 从作者原帖、许可页和权威资料回溯原始来源；找不到明确授权就标为 research-only，不把搜索结果图带进生产。

## 每次上线前的快速复核

```bash
npm run verify:content
npm run verify:launch

TAROT_PRODUCTION_ORIGIN=https://tarot-31o.pages.dev \
TAROT_REQUIRE_LLM=1 \
TAROT_REQUIRE_COUNTER=1 \
TAROT_REQUIRE_CONVERSATION_STORAGE=1 \
npm run smoke:production

TAROT_LLM_ENDPOINT=https://tarot-31o.pages.dev/api/readings/analyze \
npm run smoke:llm

TAROT_PRODUCTION_ORIGIN=https://tarot-31o.pages.dev \
npm run smoke:e2e:production
```

命令通过只代表技术链路与结构化内容门通过，不会自动补出真实用户证据、图片授权、编辑签核或支付证据。
