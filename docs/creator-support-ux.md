# 创作者支持入口：产品决策与案例

- 状态：已采纳
- 最近更新：2026-07-23
- 适用范围：MiaoTarot 的自愿支持入口、付款页面与相关文案

## 核心决策

早期实现方向基本正确，但把完整支持卡放在“结果卡片后、分享区前”，会让它稍显像广告，也会在用户完成阅读和保存结果之前打断体验。

更自然的体验顺序是：

**完整牌义 → 分享/保存 → 轻量支持入口 → 历史记录/页脚**

这一顺序遵循以下原则：

- 先完整交付价值，再邀请支持。
- 不在抽牌、翻牌和阅读牌义的仪式感过程中插入付款动作。
- 页面只展示品牌化、低视觉权重的支持入口；付款链接或二维码只在用户主动点击后出现。
- 支持是自愿行为，不影响抽牌结果，也不解锁“更准”的解读。

## 外部案例

### Lichess

Lichess 官方称平台用户每天完成超过 500 万局。捐赠入口长期存在，但只是导航和首页的次要入口，不妨碍用户下棋；进入专门的 Patron 页面后，平台才解释服务器和团队成本，并明确所有功能永久免费。

- [About Lichess](https://lichess.org/about)
- [Patron 页面](https://lichess.org/patron)

可借鉴之处：核心体验始终优先；费用用途和免费承诺在独立支持页面中讲清楚。

### itch.io

itch.io 会在用户准备下载、已经确认即将获得内容价值时，展示“按意愿付费”的支持机会，同时提供明确的 “No thanks, just take me to the downloads” 跳过方式。

- [itch.io 定价说明](https://itch.io/docs/creators/pricing)

可借鉴之处：在价值交付节点提出邀请，同时保留清晰、无压力的免费路径。

### Buy Me a Coffee

Buy Me a Coffee 建议在网站内使用按钮或组件承接支持；二维码更适合社交媒体、海报、邮件签名和线下物料。平台还鼓励创作者把 “coffee” 替换成符合自身品牌的物件，因此 MiaoTarot 使用“罐罐”作为支持隐喻是成立的。

- [网站入口与二维码建议](https://help.buymeacoffee.com/en/articles/10206530-creative-ways-to-share-your-buy-me-a-coffee-page)
- [品牌化设置](https://help.buymeacoffee.com/en/articles/10184401-how-to-set-up-your-buy-me-a-coffee-page)

可借鉴之处：站内优先使用品牌化按钮；二维码作为用户主动打开后的付款方式，而不是阅读流程中的主视觉。

### Labyrinthos

截至本次调研，Labyrinthos 在美国 App Store 有约 2.2 万条评分、4.9 分。其产品明确说明应用免费且无广告，由可选的 AI 解读、数字牌组和实体牌组购买支持。阅读产品和商店入口彼此分开，不在抽牌过程中打断用户。

- [Labyrinthos App Store 页面](https://apps.apple.com/us/app/labyrinthos-tarot-reading/id1155180220)

可借鉴之处：塔罗核心体验保持完整，把商业动作放在独立、可预期的入口中。

### 爱发电

爱发电建议创作者先说明为什么需要支持、资金会帮助什么，也允许设置“请喝咖啡、吃顿饭”式的无回馈赞助；主页链接和二维码主要用于独立主页及社交渠道。

- [爱发电创作者指南](https://guide.afdian.com/new/for-new-creators)

可借鉴之处：使用符合品牌的轻量隐喻，同时明确支持服务器、牌面制作和后续更新等具体用途。

## MiaoTarot 的落地规则

- 测试期在首页顶部常驻展示“请猫猫吃罐罐”入口，方便未完成抽牌的访客直接体验并反馈支持流程。
- 结果区先完整展示牌义和每张牌的解释。
- 分享与保存位于支持入口之前。
- 阅读后的支持入口采用单行、低对比度设计，不使用醒目的广告式卡片。
- 页脚保留常驻但低打扰的“支持 MiaoTarot”入口。
- 用户主动点击后才打开支持弹窗。
- 手机端以跳转支持页面的按钮为主；桌面端可以同时展示二维码。
- 使用“请猫猫吃罐罐”作为品牌化表达，并同时说明支持会用于服务器、牌面制作和后续更新。
- 不自动弹窗，不使用悬浮催付，不在最后一张牌翻开时出现，也不把支持和占卜准确度绑定。

当前信任边界文案：

> 支持不会影响抽牌结果，也不会解锁“更准”的解读。

## 当前支付与收入可观测性

截至 2026-07-23，页面使用 `assets/support-alipay-qr.jpg` 展示静态支付宝收款码，`SUPPORT_URL` 为空。代码没有支付 provider、订单表、金额字段、交易回调或付款成功页面。现已记录 `support_opened` 和 `support_qr_saved` 两个匿名意向事件；仍没有 `payment_started` 或 `payment_succeeded`。

因此当前能回答的是：

- **实际收到多少笔、多少钱：仓库和 Cloudflare 都无法判断。**
- **不能写成 0 元：**没有可观测记录不等于没有支付宝到账。
- **唯一可核实来源：**支付宝收款账单或用户另行维护的人工流水。

同日只读查询得到的产品信号：

| 信号 | 当前值 | 解释边界 |
| --- | ---: | --- |
| D1 `site-visits` | 15 | 由站点计数接口和 cookie 产生，不等于所有页面打开 |
| `app_opened` 匿名浏览器 | 13 | 事件窗口为 2026-07-20 至 2026-07-23 |
| 完成阅读 | 28 次 / 8 个匿名浏览器 / 19 个 session | 混有开发、smoke 与人工验收，不能视为真实用户转化 |
| LLM 请求 | 55 次，记录失败 0 次 | 测试会重复触发；成功响应数可能受版本切换影响，不能视为真实需求 |
| 分享图生成 | 6 次 / 2 个匿名浏览器 | 只能证明生成动作，不代表图片已发送或带来回流 |
| 支持弹窗打开 / 二维码保存 | 0 / 0 | 只能说明当前事件窗口没有捕获到支持意向，不等于实际收入为 0 |

新增事件可以回答“从哪个入口主动打开了支持弹窗”和“是否保存了二维码”，但不能回答用户是否扫码、付款、付款金额或退款。查询默认排除 `trafficType=internal` 的 smoke 与人工验收流量；不过至少 15 条早期 `production-smoke` 阅读开始事件没有流量标签，仍被统计为 external，所以当前窗口不适合计算转化或留存。

若以后需要验证真实支持转化，测量闭环仍需区分“看见入口 → 主动打开 → 发起付款 → provider 确认成功”，订单金额和状态由支付方回调核对；在此之前仍以支付宝账单或人工流水为准。
