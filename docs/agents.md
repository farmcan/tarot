# Codex Session 挖掘与规则沉淀

## Executive Summary

- **最强的重复信号不是某个功能，而是交付闭环。** 用户反复要求在 `main` 工作、完成后 `commit && push`、不要遗留半成品，并在长任务中持续汇报进度。
- **移动端真实体验是主要验收标准。** 多个 session 反复要求手机端端到端操作、关键状态截图、滚动与手势检查，而不是只看组件代码或桌面布局。
- **视觉生产必须先校准再批量。** 卡图应原生生成 `5:7`，先做少量样图并放进真实卡框检查；语义以 Tarot 为主，猫咪品种、梗图和情绪只能增强，不应替代牌义。
- **产品判断应以用户心理和已验证案例为依据。** 先理解用户为何愿意玩、分享、复访或支持，再研究真正有流量、收入或长期生命力的同类产品；优先复用成熟方案，避免闷头造轮子。

以上稳定规则已经写入仓库根目录的 `AGENTS.md`。一次性功能状态、尚未验证的想法和已完成任务只保留在本报告，不升级为长期规则。

## 反复出现的内容

### 1. 交付闭环反复被提醒

`commit 并 push`、`main`、`合并`、`继续`、`完成了吗` 在多个 session 中出现。它们共同说明用户要的不是建议或局部代码，而是完整结果：实现、真实验收、提交、推送，并清楚说明当前状态。

**已沉淀：** `AGENTS.md` 现在明确要求默认在 `main`，收到提交推送要求时只暂存本任务文件，完成约定验证后提交并推送；长任务按可验收切片推进并持续报告进度和阻塞。

### 2. 手机端 E2E 和截图是事实标准

手机布局混乱、分享卡截断、卡背显示不全、卡牌手势阻塞滚动、短屏看不完最后一张牌等问题反复出现。这些缺陷在组件级测试或桌面浏览器中很容易漏掉。

**已沉淀：** 继续把移动端真实入口到完成态的 E2E 作为用户可见改动的必经检查；关键状态必须截图检查；手势密集区域额外验证点击、拖动、滑动和普通页面滚动不会互相抢占。

### 3. 视觉生产要“原生比例、真实语境、小样先行”

方图放入竖卡、批量图风格漂移、二次 AIGC 丢失原始梗味、猫咪设定压过 Tarot 牌义，是最集中的视觉返工来源。用户多次要求先做一张或几张，放进卡框看过，再决定是否批量生成。

**已沉淀：** 原生 `5:7`、优先 `1020x1428`、禁止方图后裁作为常规流程；参考驱动时优先合法原始来源；Tarot 牌义是语义主线；任何批量任务先通过代表性小样验收。

### 4. 不要机械实现，要研究用户为什么愿意继续

用户持续追问：是否有人愿意玩、愿意付费、为什么会分享和复访，同类盈利产品靠什么，以及交互是否让没耐心的人觉得太重。这表明设计判断要从用户动机、时间到价值和信任感出发。

**已沉淀：** 有意义的产品与 UX 改动先写用户问题和产品假设；研究有可信流量、收入或长期运营证据的产品；区分事实与推断，并把借鉴转成 MiaoTarot 可验证的假设。

### 5. 优先复用，但要先评估是否合适

用户明确提出“能 import 就 import”，并多次要求先找库、找成熟交互和参考实现。同时，生图、支付、CDN、动画、音效等方案不能因为能接入就直接采用，还要考虑维护、性能、移动端和产品语境。

**已沉淀：** 现有 `AGENTS.md` 已要求先检查仓库依赖和既有抽象，再评估成熟包的维护、许可、安全、体积、性能和移动兼容性；只有不合适时才自建。

### 6. 核心价值先完成，AI 和支付不能抢主流程

翻完牌立即跳转、支持入口找不到、支付如何不违和、AI 成本和 fallback 等问题，本质上都在讨论价值交付顺序：用户应先完成抽牌、看到牌、获得基础解释，再进入总结、AI、分享或支持。

**已沉淀：** 抽牌、翻牌和基础解释必须在 AI、分析、分享或支付不可用时仍能完成；最后揭牌和结果必须允许用户看完；支持入口应在价值交付后清晰可见，但保持低压且不与准确度绑定。

### 7. 决策要可视化，昂贵工作先校准

卡背方案、移动布局、抽牌方式、梗图风格等决策中，用户更容易通过 HTML prototype、E2E 操作和截图比较做选择。直接描述方案或一次性批量生产会带来较大返工。

**已沉淀：** 多个可行 UX 方向先做低成本可比较原型；高成本或大批量任务先做有代表性的 pilot，并按最终验收标准在真实产品环境中检查。

## 不应写入全局规则的内容

以下内容有价值，但属于阶段性选择或具体任务，保留在 session 索引中即可：

- 某一次选定的卡背“方案 3”、某个具体标题或付款文案。
- 当前是否需要英文版、某个社交账号放在哪里、一天 100 用户的预算假设。
- 单次图片风格评价、单个图标偏好、某一批卡图的进度和耗时。
- 某个临时分支、安装步骤、API token 获取过程或当时的部署故障。
- 尚未被数据验证的增长、复访、付费和内容方向判断。

把这些写成永久指令会让未来 Agent 把旧状态误当成当前事实。若其中某项正式成为产品决策，应写入对应产品、工程或视觉契约，而不是继续堆到 `AGENTS.md`。

## 方法论优化

### A. 用“约束、决策、状态”三层记录，减少重复解释

- **约束：** 长期不轻易变化的执行边界，写入 `AGENTS.md` 或专门 contract，例如 `main`、移动 E2E、`5:7`。
- **决策：** 有上下文和取舍的产品选择，写入对应 canonical doc，并记录为什么选、为什么没选其他方案。
- **状态：** 当前做到哪里、剩余任务、临时故障和耗时，只留在 task/session 或 issue，不进入全局 memory。

### B. 复杂任务采用固定闭环

1. 明确用户问题、验收标准和不做什么。
2. 检查仓库既有实现、依赖和可复用方案。
3. 对高影响决策做有证据的外部研究。
4. 先做最小可验收切片或视觉 pilot。
5. 在真实移动旅程中 E2E 操作并截图检查。
6. 修复发现的问题，只提交本任务文件。
7. 按要求 commit、push，并交代结果、验证和剩余风险。

### C. 给批量生图增加正式门禁

批量开始前必须固定：参考来源与许可、目标牌义、prompt 版本、原生尺寸、safe area、代表性样本、卡框内截图和审批结论。任何一项变化都先重跑小样，不直接重跑全套。

### D. 让“手机端截图”服务于决策，而不只是证明页面能开

截图至少覆盖进入、操作中、关键反馈、完成态和错误／fallback。比较方案时使用同一 viewport、同一内容和同一状态；同时实际操作滚动、返回、触摸、分享和刷新，避免静态截图掩盖交互问题。

### E. 建立可量化的产品假设

对“有人愿意玩／分享／付费”的判断，不只收集审美反馈。每个重要改动应写清目标行为、最短价值路径、观测指标、失败信号和复盘时间；流量、复访、分享、支持转化分别看，不用一个指标替代全部价值。
